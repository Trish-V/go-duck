REST APIs & Search Layer

Access standard HTTP endpoints built on Gin Gonic and the Generic PostgREST-like querying capability.

Full CRUD Endpoints

Standard HTTP routing applies. Endpoints run off port 8080 naturally.

Note: Almost all routes pass through JWTMiddleware. You need an HTTP Authorization: Bearer header!

# List Entity Collection (Paginated)
GET /api/entity?page=1&pageSize=10
Authorization: Bearer <JWT>
X-Tenant-ID: default

# Fetch Single Entity
GET /api/entity/:id

# Create New Entity
POST /api/entity

# Update Existing
PUT /api/entity/:id

# Delete Existing
DELETE /api/entity/:id

Interactive Swagger UI

Instead of testing manually via cURL, the GO-DUCK generator automatically builds a swagger.json definition mapping your newly generated models and REST endpoints.

You can access the embedded, interactive Swagger API playground directly in your browser:

http://localhost:8080/swagger

Generic PostgREST RPC Engine

If your UI table wants dynamic dynamic filtering logic, calling explicit endpoints can be tedious. Instead, hit our highly performant generic Search RPC Layer that mirrors PostgREST behavior.

# Example: Finding records by explicit table name and query parameters
curl -X GET "http://localhost:8080/api/rpc/table?name=eq.John" \
  -H "Authorization: Bearer YOUR_JWT_TOKEN" \
  -H "X-Tenant-ID: tenant_B"

Operator Support Index:

  • eq: Equals (Exact Match)
  • neq: Not Equals
  • gt, gte: Greater Than, Greater Than or Equal
  • lt, lte: Less Than, Less Than or Equal
  • like: String Wildcard (Case Sensitive)
  • ilike: String Wildcard (Case Insensitive)

Wait, how secure is this? Extreme. The GORM framework natively protects the entire PostgREST system against explicit SQL Injection attempts.
